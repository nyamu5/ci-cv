# TICKET 2.6 — Analyse & Polling API Routes

**Stage:** 2 — Core Engine (PDF + Rate Limit + AI Pipeline)
**Goal:** The endpoints the frontend calls to start and track analysis.

```
Build the two API routes:

1. app/api/analyse/route.ts — POST:
   - Parse body, validate with AnalyseRequestSchema
   - Get session; build rate-limit identifier (user id or IP)
   - Call checkRateLimit — return 429 with { error, resetAt } if blocked
   - Basic injection pre-sanitisation on cv_text (strip obvious override
     phrases — the gatekeeper is the real defence, this is belt-and-braces)
   - Insert a new analyses row: status 'pending', user_id from session
     or null
   - Trigger the run-analysis task with { analysisId }
   - Return 200 { id }

2. app/api/analysis/[id]/route.ts — GET:
   - Fetch the analyses row by id
   - 404 if not found
   - Return { status, result } (result null until complete)
```

**Verification:**

```
- pnpm typecheck passes
- POST a valid CV to /api/analyse → returns an id, row appears as 'pending'
- GET /api/analysis/{id} → returns status, transitions to 'complete'
- 6 rapid POSTs from one IP → the 6th returns 429
- POST with cv_text under 200 chars → returns 400
```
